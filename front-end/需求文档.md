# 智能拍照辅助系统——需求文档

> 版本：v1.1 | 日期：2026-06-09

---

## 一、项目概述

基于 Android 的智能拍照辅助系统，在**拍照前**为用户提供主动引导。核心区别于市面所有厂商方案（Apple/华为/OPPO 等均为拍后处理），本系统在按下快门前完成四件事：

| 模块 | 英文名 | 目标 |
|------|--------|------|
| 主动寻景 | Scene Exploration | 找到当前环境中最佳拍摄机位/角度 |
| 构图指导 | Composition Guidance | 引导用户将被摄主体放到最优构图位置 |
| 姿势指导 | Pose Guidance | 实时对比用户姿势与模板，给出调整建议 |
| 语音交互 | Voice Interaction | 用户用语音表达拍摄意图，系统语音播报引导 |

---

## 三、功能需求

### 3.1 主动寻景（Scene Exploration）

#### 3.1.1 摄像头方案

**结论（来自技术调研）**：

- 双摄方案大概率不可行（OPPO/华为等限制调用，且掉帧严重）
- **推荐路径**：点击寻景按钮 → 切换超广角镜头 → 模型找到最佳场景 → 自动切回主摄并调整焦距
- **降级路径**（不支持超广角时）：主摄隐藏视角方案（内部用低焦距采图，UI显示高焦距画面）或用户旋转方案

**功能要求**：

- F-SE-01：检测当前设备是否支持超广角，动态选择寻景策略
- F-SE-02：寻景过程中实时检测场景质量并评分
- F-SE-03：找到最佳场景后自动切换至主摄，并提示用户当前机位
- F-SE-04：支持用户手动触发寻景（点击"寻景"按钮）

#### 3.1.2 最佳场景判断

采用**第三代方案 Camera Pose Adjustment**（最前沿，与产品方向最接近）：

- 输入：当前画面帧
- 输出：相机应移动的方向与幅度，如 ΔYaw=-15°, ΔPitch=+8°
- 同时支持**美学评分辅助**（NIMA-MobileNet）标注高分区域作为参考热力图

**功能要求**：

- F-SE-05：实时输出相机移动建议（方向 + 角度）
- F-SE-06：在取景框上叠加热力图或方向箭头，引导用户移动手机
- F-SE-07：达到目标构图时播放提示音/震动反馈

---

### 3.2 构图指导（Composition Guidance）

#### 3.2.1 方案选择

采用**传统 CV 检测 + 几何规则运算**（可解释性强、端侧实时运行）：

```
实时图像流 → 人体/主体检测 → 几何规则打分 → UI 文字/箭头引导
```

不采用端侧大模型美学打分（黑箱、速度慢、需大量数据）。

> 端侧美学模型（NIMA-MobileNet）保留用于**自学习功能**（见 3.4 节）。

#### 3.2.2 支持的构图方式

| 编号 | 构图方式 | 规则描述 |
|------|----------|----------|
| 1 | 三分法 | 主体落在九宫格交叉点 |
| 2 | 中心构图 | 主体居中，适合正面人像 |
| 3 | 对角线构图 | 主体沿对角线分布 |
| 4 | 框架构图 | 前景元素形成自然框架 |
| 5 | 留白构图 | 主体偏移，给视线方向留空间 |

**功能要求**：

- F-CG-01：系统根据当前人物位置与场景**自动推荐**构图方式（置于首位）
- F-CG-02：用户可手动切换其他构图方式
- F-CG-03：在取景框上叠加辅助线/箭头，告知用户需要往哪个方向移动
- F-CG-04：主体检测使用 YOLOv8n（或 NanoDet-Plus 作为轻量替代）
- F-CG-05：人脸检测使用 ML Kit Face Detection（速度最快，离线可用）

---

### 3.3 姿势指导（Pose Guidance）

#### 3.3.1 姿势模板库建设

- 模板来源：真人参考照片（高保真图）
- 关节点提取：使用 **BlazePose** 检测 33 个关键点，归一化为 0-1 坐标后存储
- 存储格式示例：
  ```json
  {
    "left_shoulder": [0.35, 0.42],
    "right_shoulder": [0.63, 0.43],
    ...
  }
  ```
- UI 展示：高保真缩略图 + 轮廓线骨架图

**功能要求**：

- F-PG-01：模板库支持离线存储，内置至少 20 组常用姿势
- F-PG-02：支持按场景分类（室外/室内/旅游景点/街拍等）
- F-PG-03：支持场景触发推荐（如 YOLO 检测到铁塔 → 推荐"手托铁塔"姿势）

#### 3.3.2 实时姿势对比

采用**角度匹配方案**（业界最常见，鲁棒性最好）：

- 实时检测：**MoveNet Lightning**（速度优先）
- 对比 8 个核心角度：

| 角度 | 使用关节 | 作用 |
|------|----------|------|
| 肘部角度 | 肩-肘-手腕 | 手臂是否伸直 |
| 肩部角度 | 肘-肩-髋 | 手臂抬高程度 |
| 膝盖角度 | 髋-膝-踝 | 是否下蹲 |
| 髋部角度 | 肩-髋-膝 | 站姿/坐姿 |
| 左右肩倾斜角 | 左肩-右肩 | 歪头/身体倾斜 |
| 左右髋倾斜角 | 左髋-右髋 | 重心偏移 |
| 身体朝向角 | 肩中心-髋中心 | 是否侧身 |
| 头部倾斜角 | 鼻子-双眼 | 回头/歪头 |

- 角度从归一化坐标实时计算，模板库中**只存坐标不存角度**
- 相似度计算辅助：余弦相似度，找出误差最大的部位，优先给出该部位的调整建议

**功能要求**：

- F-PG-04：MoveNet 实时检测帧率 ≥ 20 FPS（中端机）
- F-PG-05：每帧计算 8 个角度，与模板角度对比，输出偏差最大的 1-2 个关节
- F-PG-06：UI 以箭头 + 文字提示（如"抬高左手"）实时引导
- F-PG-07：所有角度偏差均在阈值内时，触发"姿势已到位"反馈

### 3.5 语音交互（Voice Interaction）

#### 3.5.1 模块定位

为降低用户操作成本（拍照场景下双手通常持机），引入语音交互作为输入/输出的辅助通道：

- **ASR（语音转文字）**：用户说出拍摄意图 → 系统解析 → 触发对应功能
- **TTS（文字转语音）**：系统将引导文字（如"向左移动一步"、"抬高左手"）合成语音 → 播放给用户
- **目标**：让用户无需盯屏幕也能完成拍摄过程

#### 3.5.2 技术方案：科大讯飞

| 能力 | SDK | 接入方式 | 说明 |
|------|-----|----------|------|
| 在线 ASR | 讯飞 MSC SDK / WebAPI | AAR 集成 | 识别准确率高，需联网 |
| 离线 ASR（命令词） | 讯飞离线命令词识别 | AAR + 离线词库 | 用于固定指令（如"拍照"、"换姿势"），无网可用 |
| 在线 TTS | 讯飞 MSC SDK | AAR 集成 | 多种发音人，自然度高 |
| 离线 TTS | 讯飞离线合成 | AAR + 离线发音人包 | 用于实时引导，避免网络延迟 |

**密钥管理**：APPID/APIKey/APISecret 由后端下发或编译期注入，**严禁硬编码到客户端仓库**。

#### 3.5.3 用户场景

| 场景 | 用户语音 | 系统行为 |
|------|----------|----------|
| 主动唤起寻景 | "帮我找一个好看的角度" | 触发 Scene Exploration |
| 切换构图 | "用三分法构图" | 切换至三分法 |
| 选择姿势 | "我想要一个旅游打卡的姿势" | 进入姿势库，按场景过滤推荐 |
| 实时引导反馈 | （系统播报） | "向右移动半步，抬高左手" |
| 拍照触发 | "茄子" / "拍照" | 触发快门 |

#### 3.5.4 功能要求

- **F-VI-01**：支持唤醒词触发（如"小镜小镜"）或按住按钮说话两种模式，用户可在设置中切换
- **F-VI-02**：常用拍摄指令在离线模式下也能识别（基于离线命令词库）
- **F-VI-03**：复杂语义（如自然语句描述拍摄需求）走在线 ASR + 后端意图理解
- **F-VI-04**：所有 UI 文字引导可通过 TTS 播报，用户可在设置中开启/关闭
- **F-VI-05**：TTS 在 200ms 内开始播放，避免与下一帧引导冲突；同一引导内容 5 秒内不重复播报
- **F-VI-06**：录音权限在首次进入语音功能时申请，非语音功能不强制要求该权限
- **F-VI-07**：语音指令解析失败时，UI 给出降级反馈（"没听清，请再说一次"）

#### 3.5.5 意图理解架构

```
用户说话
   ↓
讯飞 ASR (在线/离线) → 识别文本
   ↓
意图解析器 (Intent Parser)
   ├── 离线规则匹配（命令词 → 动作）
   └── 在线 LLM/NLU（复杂语句 → 结构化意图）   ← 二期
   ↓
Action Dispatcher → 调用对应功能模块
```

**意图结构示例**：
```kotlin
data class VoiceIntent(
    val type: IntentType,            // SHOOT / CHANGE_POSE / EXPLORE / ...
    val params: Map<String, String>, // {"compositionMode": "rule_of_thirds"}
    val rawText: String,
    val confidence: Float
)
```

#### 3.5.6 后端依赖

- 在线意图理解 API（二期）：客户端把识别文本发给后端，后端返回结构化意图
- 接口由 Retrofit 调用，详见 **第六章 数据与接口设计**（待补）

## 四、技术选型

### 4.1 推理引擎

| 引擎 | 推荐场景 | 说明 |
|------|----------|------|
| **MNN**（首选） | 所有端侧模型 | 阿里出品，轻量高效，支持 GPU 加速（OpenCL），支持 ONNX 转换 |
| TFLite | MoveNet / BlazePose | Google 官方，MoveNet 原生支持，无需转换 |
| NCNN | 备选 | 腾讯出品，CPU 性能优秀，兼容性好 |

**MNN 集成要点**：
- 模型转换路径：原始模型 → ONNX（中间格式）→ MNN 格式 + 量化压缩
- 动态库：将 `libMNN.so` 放入 `src/main/jniLibs/arm64-v8a/`
- 图像预处理优化：用 MNN 内置 `ImageProcess` 模块替代 OpenCV 做 Resize/Crop/YUV→RGB，可减少 40-60% 预处理耗时
- 对实时性要求高的模型（MoveNet）开启 `MNN_FORWARD_OPENCL` 使用 GPU 推理

### 4.2 各模块模型选型

| 模块 | 任务 | 推荐模型 | 备选 | 理由 |
|------|------|----------|------|------|
| Scene Exploration | 场景目标检测 | **YOLOv8n** | NanoDet-Plus | 最轻量，适合实时检测 |
| Scene Exploration | 美学评分/热力图 | **NIMA-MobileNet** | EfficientNet-Lite0 | 轻量，业界美学评分标准模型 |
| Scene Exploration | 语义分割（备用） | **MobileSAM** | MobileNetV3 | SAM 轻量版，实时可用 |
| Composition Guidance | 主体检测 | **YOLOv8n** | NanoDet-Plus | 复用寻景模块，减少模型加载 |
| Composition Guidance | 人脸检测 | **ML Kit Face Detection** | — | 无需额外部署，离线 SDK |
| Pose Guidance | 模板关节点提取（离线） | **BlazePose** | MoveNet Thunder | 精度高，33点，用于模板库建设 |
| Pose Guidance | 实时关节点检测 | **MoveNet Lightning** | MoveNet Thunder | 速度最快，17点足够角度计算 |
| 图像传统处理 | Resize/色彩空间等 | **OpenCV** | MNN ImageProcess | 通用，成熟稳定 |

### 4.3 模型对比说明

**MoveNet Lightning vs Thunder**：
- Lightning：速度优先，~6ms/帧（骁龙8系），适合实时预览
- Thunder：精度优先，~35ms/帧，适合静态图分析
- **本项目实时指导用 Lightning，模板库精标可用 Thunder**

**YOLOv8n vs NanoDet-Plus**：
- YOLOv8n：生态更好，ONNX 导出成熟，精度略高
- NanoDet-Plus：模型更小（~1MB），极低端设备首选
- **推荐 YOLOv8n，NanoDet-Plus 作为低端机降级方案**

**MobileNetV3 / EfficientNet-Lite0**：
- 用于场景分类（室内/室外/景区等），辅助构图推荐
- EfficientNet-Lite0 精度略高，MobileNetV3 速度略快
- **可复用于场景分类 + 构图推荐，推荐 EfficientNet-Lite0**

## 七、开发规范

### 7.1 Compose 状态管理（UI / UiState / ViewModel 三层）

> 这是 Google 官方推荐的 **MVVM + UDF（Unidirectional Data Flow，单向数据流）** 模式，本项目**强制采用**。

#### 7.1.1 三层职责

```
┌─────────────────────────────────────────────────────────┐
│  UI 层 (@Composable)                                    │
│  • 无状态（stateless）：所有数据从参数传入              │
│  • 不直接持有可变状态；不直接调用业务/IO                │
│  • 通过回调（lambda）把用户事件抛给 ViewModel           │
└──────────────────────────┬──────────────────────────────┘
            读 ▲                     ▼ 派发事件
┌─────────────────────────────────────────────────────────┐
│  UiState (data class)                                   │
│  • 不可变（immutable）的屏幕状态快照                    │
│  • 单一可信源（Single Source of Truth）                 │
│  • 例：data class CameraUiState(                        │
│         val isExploring: Boolean = false,               │
│         val composition: CompositionMode = ...,         │
│         val poseHint: String? = null,                   │
│         val errorMessage: String? = null                │
│      )                                                  │
└──────────────────────────▲──────────────────────────────┘
                  暴露 StateFlow │
┌─────────────────────────────────────────────────────────┐
│  ViewModel (业务编排层)                                 │
│  • 持有 _uiState: MutableStateFlow<UiState>             │
│  • 暴露 uiState: StateFlow<UiState>                     │
│  • 接收 UI Intent（onShootClick / onModeChange ...）    │
│  • 调用 Repository / UseCase，更新 UiState              │
│  • 不持有 Context、View、Composable 引用                │
└─────────────────────────────────────────────────────────┘
```

#### 7.1.2 模式分析（结论：✅ 采用）

**优点**：
- **UI 是状态的纯函数**（`f(state) → UI`），易预览、易测试、易复现 bug
- **单一可信源**，避免状态散落到多个 Composable 的 `remember { mutableStateOf(...) }` 中
- **生命周期安全**：`StateFlow` + `collectAsStateWithLifecycle()` 自动随生命周期挂起/恢复
- **天然契合本项目**：CameraX 输出帧流（Flow）、姿势检测结果流、语音事件流都能通过 Flow 操作符聚合到 UiState
- **协作友好**：UiState 是契约，前端写 UI、另一个人写 ViewModel，可以并行（先约定 UiState 再开工）

**注意事项（本项目专属）**：
- **重对象不进 UiState**：`Bitmap` / 相机 `ImageProxy` / 推理张量等不要塞进 UiState；用单独的 `SharedFlow<Bitmap>` 或 `Channel` 传递，UI 侧 `collectAsState` 或直接渲染
- **高频更新降噪**：实时帧（20+ FPS）走 `Flow.conflate()` + `distinctUntilChanged()`，避免 UiState 抖动导致重组风暴
- **Compose 跳过优化**：UiState 必须是 `@Stable` 或 `@Immutable` 数据类，列表用 `ImmutableList`（kotlinx.collections.immutable）
- **错误用一次性事件**：Toast / Snackbar 类一次性消息用 `Channel<UiEvent>` 或 `SharedFlow`，**不要**塞 UiState（否则配置变更会重复显示）

## 八、多人协作开发指南

### 8.1 协作哲学（先回答你的问题）

> **"应该先搭好文件框架吗？"** → **是的，必须先搭。**

理由：
- 多人项目最大的成本不是写代码，是 **合并冲突 + 风格漂移 + 重复造轮子**
- 框架先行可以让每个人在自己的目录里独立开发，**冲突域局部化**
- 接口先行（先定义模块间契约）让前后端、上下游可以**并行**而不是串行

**核心原则**：
1. **先约定，再开发**：包结构、UiState、模块接口先定下来，再分头实现
2. **一人一模块**：每个 feature module 主负责人明确，其他人改前先沟通
3. **接口隔离**：模块间只通过 interface 通信，不直接依赖实现类
4. **小步快跑**：每个 PR 控制在 400 行内，1-2 天合一次主干

### 8.2 推荐项目结构（单模块，包路径隔离）

**约定 → 实现 → 集成** 三步走，比"先实现再凑接口"快 2 倍。

### 8.4 Git 工作流

**采用 GitHub Flow（推荐，简单）**：

```
main (受保护，CI 通过才能合)
  ↑
  └── feature/pose-realtime-compare      ← C 在写
  └── feature/voice-asr-integration      ← D 在写
  └── fix/camera-focus-on-mi10           ← Hotfix
```

规则：
1. **主分支保护**：禁止直接 push 到 `main`，必须 PR
2. **分支命名**：`feature/<scope>-<desc>` / `fix/<scope>-<desc>` / `chore/<desc>`
3. **PR 要求**：
   - 标题：`[模块] 一句话描述`，如 `[pose] 实时角度对比 MVP`
   - 描述：变更点、测试方式、关联需求条目（如 F-PG-04）
   - 至少 1 人 review，CI 全绿
   - 控制在 ~400 行以内，超过拆分
4. **Commit 信息**：Conventional Commits（`feat: ...` / `fix: ...` / `refactor: ...`）
5. **冲突处理**：本地 `git rebase main` 解决，不要直接 merge main 到 feature 分支制造蛇形历史
6. **Squash merge** 入 main，保持历史线性

### 8.7 防止"代码乱"的硬规则

3. **CI 卡住三件套**：编译通过 / ktlint 通过 / 单测通过，缺一不能合

6. **共享代码下沉**：第二次出现的工具函数立刻提到 `core/common`，不要复制粘贴第三次
